## 1. สรุปผู้บริหาร

สร้าง **docbrief** = โมดูลจัดการเอกสารการเงินบน AI OpsBrief OS เดิม เสร็จและใช้งานจริงบน production ภายใน 1 วัน

```
📄 อัปโหลดบิล → 🤖 Claude อ่าน → ✅ ตรวจกฎไทย → 👤 owner ยืนยัน → 📊 Google Sheet
```

**สิ่งสำคัญที่สุดของรายงานนี้ไม่ใช่ว่าสร้างอะไรได้ แต่คือ — สเปกที่ล็อกไว้ตอนเช้าไม่ตรงกับธุรกิจจริง และเราค้นพบเพราะเอาเอกสารจริงมาทดสอบ**

| | |
|---|---|
| โค้ดใหม่ | 7 ไฟล์ · แตะโมดูลเดิม 3 ไฟล์ (เมนู 1 บรรทัด + Daily Brief) |
| ตารางใหม่ | 4 ตาราง prefix `doc_` · additive ล้วน · rollback ได้ |
| QA | intake 30/30 · validation 27/27 · end-to-end กับเอกสารจริง ✅ |
| ต้นทุน | ~1.36 บาท/ใบ (เพดานอ่อน 2 บาท) |
| Dependency ใหม่ | **0** |

## 2. เรื่องที่ทีมต้องรู้ก่อนอย่างอื่น — สเปกเดิมผิด

### 2.1 สิ่งที่เราเชื่อตอนเช้า
สเปก Phase 0 ล็อกไว้ว่า V1 = **"บิลซื้อจาก vendor"** (invoice / receipt / ใบกำกับภาษี ฝั่งซื้อ)
วางกฎ validation, idempotency key, field spec ทั้งหมดบนสมมติฐานนี้

### 2.2 สิ่งที่พบเมื่อขอเอกสารจริง

| รอบ | ได้เอกสารอะไร | ผล |
|-----|--------------|-----|
| 1 | Screenshot Excel — **บิลขายที่ร้านออกเอง** | ผิด scope · ไม่มีเลขที่เอกสาร |
| 2 | รูปถ่าย — **ใบเสร็จภาษี ภ.พ.30 กรมสรรพากร** | ไม่ใช่บิลซื้อ · ไม่แยก VAT |
| 3 | รูปถ่าย — **สมุดจดลายมือภาษาจีน** | รายการอะไหล่เขียนมือ |

**คำตอบจาก owner:** *"อะไหล่นำเข้าทั้งหมด ซื้อเหมาทั้งตู้ 7 แสน–1 ล้านบาท **ไม่มีใบเสร็จรับเงิน**"*

### 2.3 บทสรุป
> **"บิลซื้อจาก vendor" แบบที่สเปกวางไว้ ไม่มีอยู่จริงในธุรกิจนี้**
> เกือบสร้างระบบแก้ปัญหาที่ไม่มีอยู่

### 2.4 pain จริงที่ owner ตอบเอง
**อันดับ 1 = คีย์สต็อกอะไหล่เข้าระบบ** (และต้องทำบัญชีด้วย)

### 2.5 สิ่งที่แก้
แบ่งเป็น 2 โปรไฟล์ ใช้เครื่องยนต์เดียวกัน:

| | **A. สต็อก** | **B. บัญชี** ← ทำก่อน |
|---|---|---|
| เอกสาร | สมุดจดจีน / packing list | ใบเสร็จศุลกากร · ภ.พ.30 · ค่าใช้จ่าย |
| ดึง | รายการหลายบรรทัด | หัวเอกสาร |
| ปลายทาง | Sheet → `sync-stock` เดิม | Sheet บัญชี |
| สถานะ | สำรวจแล้ว ยังไม่สร้าง | **เสร็จ ใช้งานจริง** |

**ขอให้ทีมตรวจ:** การตัดสินใจทำ B ก่อน A ทั้งที่ A คือ pain อันดับ 1 — เหตุผลคือ B เสร็จไป 90% แล้ว ปิดให้ใช้งานได้เร็ว แล้วค่อยลุย A ที่ใหญ่กว่า **เห็นด้วยหรือควรกลับลำดับ?**

## 4. บันทึกการตัดสินใจ (ให้ทีมตรวจ)

| # | ตัดสินใจอะไร | เหตุผล | ทีมควรตรวจ |
|---|-------------|--------|------------|
| 1 | ใช้ **Supabase เดิม** ไม่สร้าง DB ใหม่ | ห้ามสร้าง infra ซ้ำ | ✅ |
| 2 | ใช้ **raw fetch** เรียก Claude ไม่ใช้ SDK | ตาม pattern เดิมใน repo · ไม่เพิ่ม dependency | ⚠️ แลกกับ type safety + retry อัตโนมัติ |
| 3 | **Apps Script Web App** แทน Google Sheets API | ไม่ต้องลง package / service account / OAuth | ⚠️ ผูกกับบัญชี Google ของ owner |
| 4 | Google Sheet = **ปลายทางสุดท้าย** ไม่เขียนเข้า `ledger` | core มี finance stack อยู่แล้ว ห้ามทำซ้ำ | ✅ |
| 5 | `subtotal`/`vat` เปลี่ยนเป็น **optional** | ใบเสร็จศุลกากร/ภ.พ.30 ไม่แยก VAT | ✅ แก้สเปก §3 แล้ว |
| 6 | **ไม่เชื่อ confidence ของโมเดล** ใช้กฎคณิตแทน | ทดสอบพบโมเดลรายงาน 0.44 แต่ค่าถูกจริง | ⭐ **ข้อสำคัญ** |
| 7 | อ่านเอกสาร **ต้องกดเอง** ไม่อัตโนมัติ | คุมต้นทุนให้เห็นชัด | ⚠️ แลกกับความสะดวก |
| 8 | Daily Brief เรียก function ตรง ไม่ผ่าน HTTP | อยู่แอปเดียวกัน เร็วกว่า ไม่เปิดช่อง auth เพิ่ม | ✅ endpoint ยังมีตามสเปก |

### ขยายข้อ 6 — ข้อค้นพบที่กระทบสเปก
สเปกเดิม §3 กำหนด *"confidence < 0.85 → ตีธงให้คนตรวจ"*

ทดสอบสมุดจดลายมือจีน: โมเดลรายงาน confidence **เฉลี่ย 0.44** ทุกบรรทัด → **owner ยืนยันว่าค่าที่ดึงมาถูกต้อง**

**ถ้าใช้กฎเดิม = ตีธงทุกบรรทัด ธงจะไร้ความหมาย**

**แก้เป็นลำดับความน่าเชื่อถือ:**
1. **กฎคณิต** (`subtotal + vat = total`, `qty × price = amount`) ← เชื่อได้สูงสุด
2. **confidence** ← ใช้เฉพาะเมื่อกฎข้อ 1 รันไม่ได้

> เลขที่บวก/คูณลงตัว = อ่านถูกแน่นอน โดยไม่ต้องเชื่อความเห็นของโมเดล

## 5. หลักฐาน QA

### 5.1 Phase 1 Intake — 30/30 ✅
ทดสอบกับไฟล์จริง 9 แบบ (PDF 1/3/6 หน้า, PNG, JPG, .txt, PDF เสีย, 11 MB, ไฟล์ซ้ำ)

- สร้าง record ทุกครั้งแม้ validate ไม่ผ่าน ✅
- ต้นฉบับไม่หาย · ไฟล์ซ้ำไม่เขียนทับ ✅
- **audit ลบ/แก้ไม่ได้จริง** — ลอง UPDATE/DELETE แล้ว error ทั้งคู่ ✅
- ตารางเดิมของ core อ่านได้ปกติ ✅
- ลบข้อมูลทดสอบหมดแล้ว

### 5.2 Phase 2 Parse — ทดสอบเอกสารจริง
| เอกสาร | ผล |
|--------|-----|
| ใบเสร็จ ภ.พ.30 × 3 | อ่านเลขที่/วันที่/ยอดถูกทั้งหมด · **แปลง พ.ศ.→ค.ศ. ถูก** |
| บิล Excel × 3 | VAT 7% + arithmetic ตรงทุกใบ |
| สมุดจดจีน | ดึง 19 บรรทัด · แปลชื่ออะไหล่จีน→ไทยได้ 18/19 |

**ไม่มี hallucination แม้แต่ครั้งเดียว** — ช่องว่างในบิล → คืน `null` + confidence 0 พร้อมเหตุผล ไม่แต่งตัวเลข

**แยกผู้ออกบิลกับผู้จ่ายได้ถูก** — เจอเลข 13 หลักในใบเสร็จภาษี แต่ไม่ยัดลง `vendor_tax_id` เพราะรู้ว่าเป็นเลขของผู้จ่าย

**ต้นทุน:** 1.36 บาท/ใบ (input ~5,010 tok / output ~555 tok)

### 5.3 Phase 3 Validate — 27/27 ✅
- checksum เลขภาษี: ทดสอบด้วย **เลขจริงจากใบเสร็จศุลกากร** ผ่าน · แก้ผิด 1 หลักตกทันที
- arithmetic / VAT 7% / tolerance 1 สตางค์ ✅
- ใบเสร็จศุลกากร (ไม่มี subtotal/vat) → **ไม่มีธงเลย** ✅
- confidence 0.4 + เลขลงตัว → **ไม่ตีธง** ✅ (ข้อ 6)
- export key deterministic · ขีดคั่นไม่ทำให้เปลี่ยน ✅

### 5.4 Phase 4 Export — พิสูจน์บน production
```
Sheet แถว 6:  กองบริหารการคลังและรายได้ กรมสรรพากร
              67130148302 · 2024-09-21 · 3,640 THB
doc_exports:  status = success
audit:        pending_review → confirmed → exporting → exported
```
**ตรงกันทั้ง 3 ที่**

## 9. คำถามที่อยากให้ทีมตอบ

1. **ลำดับ B ก่อน A ถูกไหม?** — A (สต็อก) คือ pain อันดับ 1 แต่เราทำ B (บัญชี) ก่อนเพราะเสร็จไป 90%
2. **ข้อ 6 (ไม่เชื่อ confidence) เห็นด้วยไหม?** — เป็นการแก้สเปกที่มาจากข้อมูลจริง
3. **การ์ดเอกสารใน Daily Brief อยู่บนสุด** — เด่นเกินไปจนแย่งความสนใจจาก "วันนี้ทำอะไรก่อน" หรือเปล่า?
4. **ความเสี่ยงข้อ 1 และ 2** ยอมรับได้ไหม หรือควรแก้ก่อนขยาย
5. **ควรวัด accuracy ให้ผ่าน §6 ก่อนไหม** หรือให้ pilot ตัดสินพอ
